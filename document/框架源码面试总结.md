## 一、Spring Bean生命周期

### 1.1 整体流程图

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Spring Bean 生命周期                          │
├─────────────────────────────────────────────────────────────────────┤
│  1. 实例化(Instantiation)                                           │
│     └─> 反射调用构造方法，创建Bean实例                                │
│                                                                     │
│  2. 属性赋值(Populate)                                              │
│     └─> 依赖注入、@Autowired注入                                     │
│                                                                     │
│  3. 初始化(Initialization)                                          │
│     └─> Aware回调 → @PostConstruct → InitializingBean → init-method │
│                                                                     │
│  4. 销毁(Destruction)                                               │
│     └─> @PreDestroy → DisposableBean → destroy-method              │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 源码流程

```java
// AbstractAutowireCapableBeanFactory.java

// 1. 实例化前 - 可返回自定义Bean（短路操作，若返回非null则跳过后续实例化）
postProcessBeforeInstantiation()

// 2. 实例化 - 反射创建Bean
createBeanInstance()

// 3. 实例化后 - 收集@Autowire等元信息
postProcessMergedBeanDefinition()

// 4. 属性填充
populateBean()

// 5. 初始化前
postProcessBeforeInitialization()
  → ApplicationContextAware#setApplicationContext()
  → @PostConstruct  // 通过InitDestroyAnnotationBeanPostProcessor

// 6. 初始化
invokeAwareMethods()  // BeanNameAware, BeanFactoryAware等
invokeInitMethods()
  → InitializingBean#afterPropertiesSet()
  → 自定义init-method

// 7. 初始化后 - AOP代理创建入口
postProcessAfterInitialization()
  → AbstractAdvisorAutoProxyCreator 创建代理

// 8. 销毁
destroy()
  → @PreDestroy
  → DisposableBean#destroy()
  → 自定义destroy-method
```

> ⚠️ **踩坑细节**：`invokeAwareMethods()` 只处理 `BeanNameAware`、`BeanClassLoaderAware`、`BeanFactoryAware` 三个。`ApplicationContextAware` 是由 `ApplicationContextAwareProcessor`（一个 BeanPostProcessor）在 `postProcessBeforeInitialization` 阶段处理的，**不在** `invokeAwareMethods` 里，这是常见混淆点。

### 1.3 初始化回调的完整执行顺序（细化）

```
Aware 接口回调（BeanNameAware → BeanClassLoaderAware → BeanFactoryAware）
    ↓
postProcessBeforeInitialization（含 ApplicationContextAware、@PostConstruct）
    ↓
InitializingBean#afterPropertiesSet()
    ↓
自定义 init-method（xml 的 init-method 或 @Bean(initMethod="...")）
    ↓
postProcessAfterInitialization（AOP 代理在此生成）
```

**面试追问：`@PostConstruct` 和 `afterPropertiesSet` 哪个先执行？**  
→ `@PostConstruct` 先。因为它由 `InitDestroyAnnotationBeanPostProcessor` 在 `postProcessBeforeInitialization` 阶段处理，而 `afterPropertiesSet` 属于 `invokeInitMethods` 阶段，后者更晚。

### 1.4 常见面试题

**Q: BeanPostProcessor 和 BeanFactoryPostProcessor 区别？**

| 类型 | 作用 | 执行时机 |
|------|------|----------|
| BeanFactoryPostProcessor | 处理BeanDefinition | Bean创建前 |
| BeanPostProcessor | 处理Bean实例 | Bean创建时（初始化前后） |

> **追问：能举一个 BeanFactoryPostProcessor 的实际应用吗？**  
> → `ConfigurationClassPostProcessor` 是最重要的实现，负责解析 `@Configuration`、`@ComponentScan`、`@Import`、`@Bean` 等注解，将 BeanDefinition 注册到容器。`PropertyPlaceholderConfigurer` 负责处理 `${...}` 占位符替换，也是典型实现。

```java
// 示例：自定义BeanPostProcessor
@Component
public class MyBeanPostProcessor implements BeanPostProcessor {
    @Override
    public Object postProcessBeforeInitialization(Object bean, String beanName) {
        System.out.println("初始化前: " + beanName);
        return bean;
    }
    
    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        System.out.println("初始化后: " + beanName);
        return bean;
    }
}
```

> **追问：BeanPostProcessor 本身是什么时候注册的？**  
> → 在 `refresh()` 的第 6 步 `registerBeanPostProcessors()` 中注册，早于普通 Bean 的创建（第 11 步）。这意味着 BeanPostProcessor 自身的依赖不能依赖普通 Bean，否则会导致那些普通 Bean 提前实例化，跳过 BeanPostProcessor 处理。

## 二、Spring IOC容器加载流程

### 2.1 refresh() 核心步骤

```java
// AbstractApplicationContext.refresh()
public void refresh() throws BeansException {
    // 1. 准备环境
    prepareRefresh();
    
    // 2. 创建BeanFactory
    ConfigurableListableBeanFactory beanFactory = obtainFreshBeanFactory();
    
    // 3. 准备BeanFactory
    prepareBeanFactory(beanFactory);
    
    // 4. 后置处理BeanFactory
    postProcessBeanFactory(beanFactory);
    
    // 5. 执行BeanFactoryPostProcessor
    invokeBeanFactoryPostProcessors(beanFactory);
    
    // 6. 注册BeanPostProcessor
    registerBeanPostProcessors(beanFactory);
    
    // 7. 初始化MessageSource
    initMessageSource();
    
    // 8. 初始化事件广播器
    initApplicationEventMulticaster();
    
    // 9. 子类初始化
    onRefresh();
    
    // 10. 注册监听器
    registerListeners();
    
    // 11. 创建所有单例Bean ★★★
    finishBeanFactoryInitialization(beanFactory);
    
    // 12. 完成刷新
    finishRefresh();
}
```

### 2.2 各步骤深化说明

| 步骤 | 方法 | 核心干了什么 |
|------|------|------------|
| 1 | `prepareRefresh()` | 记录启动时间、设置 active 标志、初始化 Environment 中的 PropertySource |
| 2 | `obtainFreshBeanFactory()` | XML方式：解析xml创建 BeanDefinition；注解方式：返回已有 BeanFactory |
| 3 | `prepareBeanFactory()` | 注册 ClassLoader、BeanExpressionResolver（SpEL）、内置 BeanPostProcessor（如 `ApplicationContextAwareProcessor`） |
| 5 | `invokeBeanFactoryPostProcessors()` | **先执行 `BeanDefinitionRegistryPostProcessor`（含 `ConfigurationClassPostProcessor`），再执行 `BeanFactoryPostProcessor`** |
| 6 | `registerBeanPostProcessors()` | 按 PriorityOrdered → Ordered → 普通 的顺序注册 |
| 11 | `finishBeanFactoryInitialization()` | 遍历所有非懒加载单例 BeanDefinition，调用 `getBean()` 创建 |
| 12 | `finishRefresh()` | 发布 `ContextRefreshedEvent`，启动 Lifecycle Bean |

> **追问：`invokeBeanFactoryPostProcessors` 中 `BeanDefinitionRegistryPostProcessor` 为什么要先执行？**  
> → 因为它可以动态注册新的 BeanDefinition（比如 `ConfigurationClassPostProcessor` 扫描 `@Component`），必须在所有 `BeanFactoryPostProcessor` 修改 BeanDefinition 之前把定义注册完整。

### 2.3 BeanDefinition 注册流程

```
扫描包路径 (@ComponentScan)
    ↓
ClassPathBeanDefinitionScanner.scan()
    ↓
MetadataReader 解析 .class 文件 (ASM技术，不加载Class，性能更好)
    ↓
excludeFilters / includeFilters 过滤
    ↓
@Conditional 条件注解判断
    ↓
ScannedGenericBeanDefinition
    ↓
BeanDefinitionRegistry.registerBeanDefinition()
```

> **追问：为什么用 ASM 而不是反射读取类元数据？**  
> → ASM 直接读取字节码，不需要将类加载到 JVM，避免了类加载带来的内存开销和潜在的类加载器问题，在大规模扫描时性能远优于反射。

### 2.4 BeanDefinition 核心属性

```java
// BeanDefinition 关键信息
interface BeanDefinition {
    String getBeanClassName();          // 全限定类名
    String getScope();                  // singleton / prototype
    boolean isLazyInit();               // 是否懒加载
    String[] getDependsOn();            // @DependsOn 依赖
    ConstructorArgumentValues getConstructorArgumentValues();
    MutablePropertyValues getPropertyValues();
```

## 四、Spring 循环依赖

### 4.1 什么是循环依赖

```
A → B → C → A  (构造器循环依赖 - 无法解决)
A → B → A      (setter循环依赖 - 可解决)
```

### 4.2 三级缓存详解

```java
// DefaultSingletonBeanRegistry.java

// ★ 一级缓存：完整Bean，直接可用
private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);

// ★ 二级缓存：早期Bean，未完成属性填充
private final Map<String, Object> earlySingletonObjects = new HashMap<>(16);

// ★ 三级缓存：ObjectFactory，Lambda表达式
private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>(16);
```

### 4.3 解决流程图

```
┌────────────────────────────────────────────────────────────────┐
│                   Spring 循环依赖解决流程                        │
├────────────────────────────────────────────────────────────────┤
│                                                                │
│  getBean("A")                                                  │
│       ↓                                                        │
│  创建 A → 放入三级缓存 → 填充属性发现需要B                       │
│       ↓                                                        │
│  getBean("B")                                                  │
│       ↓                                                        │
│  创建 B → 放入三级缓存 → 填充属性发现需要A                       │
│       ↓                                                        │
│  getBean("A") → 三级缓存有A! → 升到二级缓存 → 注入B              │
│       ↓                                                        │
│  B创建完成 → 放入一级缓存 → 返回继续填充A                        │
│       ↓                                                        │
│  A创建完成 → 放入一级缓存 ✓                                     │
│                                                                │
└────────────────────────────────────────────────────────────────┘
```

### 4.4 源码核心逻辑

```java
// getSingleton() 三级缓存查询
protected Object getSingleton(String beanName, boolean allowEarlyReference) {
    // 1. 查一级缓存
    Object singletonObject = this.singletonObjects.get(beanName);
    if (singletonObject == null && isSingletonCurrentlyInCreation(beanName)) {
        // 2. 查二级缓存
        singletonObject = this.earlySingletonObjects.get(beanName);
        if (singletonObject == null && allowEarlyReference) {
            // 3. 查三级缓存，通过ObjectFactory获取早期引用
            ObjectFactory<?> singletonFactory = this.singletonFactories.get(beanName);
            if (singletonFactory != null) {
                singletonObject = singletonFactory.getObject();
                // 升级到二级缓存
                this.earlySingletonObjects.put(beanName, singletonObject);
                this.singletonFactories.remove(beanName);
            }
        }
    }
    return singletonObject;
}
```

### 4.5 三级缓存中 ObjectFactory 的作用（重点追问）

> **追问：二级缓存能解决循环依赖，为什么还需要三级缓存？**

三级缓存存的是 `ObjectFactory`（一个 Lambda），其 `getObject()` 实际调用的是：

```java
// AbstractAutowireCapableBeanFactory
addSingletonFactory(beanName, () -> getEarlyBeanReference(beanName, mbd, bean));

// getEarlyBeanReference 会触发 SmartInstantiationAwareBeanPostProcessor
protected Object getEarlyBeanReference(String beanName, RootBeanDefinition mbd, Object bean) {
    Object exposedObject = bean;
    for (BeanPostProcessor bp : getBeanPostProcessors()) {
        if (bp instanceof SmartInstantiationAwareBeanPostProcessor) {
            exposedObject = ((SmartInstantiationAwareBeanPostProcessor) bp)
                .getEarlyBeanReference(exposedObject, beanName);
        }
    }
    return exposedObject;
}
```

**关键点**：如果 Bean 需要 AOP 代理，`getEarlyBeanReference` 会提前生成代理对象。如果只用二级缓存直接存原始 Bean，当 Bean 有 AOP 时，其他 Bean 注入的是原始对象，而最终放入一级缓存的是代理对象，**两者不一致**，会导致问题。三级缓存通过延迟调用 ObjectFactory，确保拿到的始终是最终形态（可能是代理）的早期引用。

### 4.6 无法解决的情况

| 场景 | 原因 |
|------|------|
| 构造器循环依赖 | 构造时就需要参数，无法提前暴露 |
| prototype 循环依赖 | 非单例，不缓存 |
| @Async 循环依赖 | 代理对象导致引用变化（`@Async` 在后置处理器中生成代理，与早期引用不一致） |

> **追问：`@Async` 为什么会破坏循环依赖？**  
> → `@Async` 由 `AsyncAnnotationBeanPostProcessor` 处理，它在 `postProcessAfterInitialization` 阶段返回一个**新的**代理对象，而此时二级缓存里存的是旧的早期引用（原始对象或普通 AOP 代理）。Spring 检测到最终 Bean 与早期暴露的引用不一致，会抛出异常。解决方案是在 `@Async` Bean 上加 `@Lazy` 打破依赖，或提取接口让注入方持有接口引用。

## 五、Spring 推断构造方法

### 5.1 推断规则

```java
// AutowiredAnnotationBeanPostProcessor
// 1. 只有一个构造方法 → 使用该构造方法
// 2. 有多个构造方法：
//    - 有 @Autowired 注解 → 使用该构造方法
//    - 有无参构造 → 使用无参构造
//    - 多个有参构造 → 根据参数从容器匹配
```

### 5.2 推断规则细化（各情形汇总）

| 构造方法情况 | 推断结果 |
|-------------|---------|
| 只有一个无参构造 | 使用无参构造 |
| 只有一个有参构造（无 `@Autowired`） | **直接使用该有参构造**（Spring 会尝试从容器注入参数） |
| 多个构造方法，其中一个有 `@Autowired` | 使用有 `@Autowired` 的那个 |
| 多个构造方法，多个都有 `@Autowired` | 抛出异常（只能有一个 `@Autowired(required=true)`） |
| 多个构造方法，有一个无参 + 多个有参（无注解） | 使用无参构造 |
| 多个构造方法，全部有参（无注解、无无参） | 抛出异常，无法推断 |

> **追问：`@Autowired(required=false)` 加在多个构造方法上会怎样？**  
> → Spring 会收集所有 `required=false` 的候选构造方法，按参数数量从多到少排列，逐一尝试从容器匹配参数，选取可以完全匹配的那个。如果都能匹配，选参数最多的（贪心策略）。

## 六、Spring AOP

### 6.1 核心概念

| 概念 | 说明 |
|------|------|
| Aspect | 切面（类） |
| JoinPoint | 连接点（方法） |
| Pointcut | 切点（匹配规则） |
| Advice | 通知（增强逻辑） |
| Weaving | 织入 |
| Target | 目标对象 |
| Proxy | 代理对象 |

### 6.2 Advice 执行顺序

```java
@Aspect
@Component
public class LogAspect {
    
    @Before("execution(* com.demo.*.*(..))")
    public void before(JoinPoint joinPoint) {
        System.out.println("1. @Before 执行");
    }
    
    @After("execution(* com.demo.*.*(..))")
    public void after(JoinPoint joinPoint) {
        System.out.println("3. @After 执行");
    }
    
    @AfterReturning("execution(* com.demo.*.*(..))")
    public void afterReturning(JoinPoint joinPoint) {
        System.out.println("4. @AfterReturning 执行");
    }
    
    @Around("execution(* com.demo.*.*(..))")
    public Object around(ProceedingJoinPoint pjp) throws Throwable {
        System.out.println("0. @Around 前置");
        Object result = pjp.proceed();
        System.out.println("5. @Around 后置");
        return result;
    }
}
```

### 6.3 Advice 完整执行顺序（细化）

**正常执行（无异常）：**
```
@Around 前置 → @Before → 目标方法 → @Around 后置 → @After → @AfterReturning
```

**抛出异常：**
```
@Around 前置 → @Before → 目标方法(异常) → @After → @AfterThrows
（@Around 后置 和 @AfterReturning 不执行）
```

> **追问：多个切面同时作用于一个方法，执行顺序如何？**  
> → 由 `@Order` 注解或实现 `Ordered` 接口控制。Order 值越小优先级越高，类似过滤器链：**高优先级切面的 `@Before` 先执行，其 `@After`/`@AfterReturning` 后执行**（洋葱模型）。

```
切面A(@Order(1)) 外层
  切面B(@Order(2)) 内层
    目标方法
  切面B 后置
切面A 后置
```

### 6.4 代理方式选择

```
┌──────────────────────────────────────────┐
│           代理方式选择逻辑                 │
├──────────────────────────────────────────┤
│  目标类实现接口?                          │
│       ↓                                  │
│   Yes ↓ No                               │
│   ┌────┴────┐                            │
│   ↓          ↓                           │
│  JDK代理    CGLIB代理                     │
│  (Proxy)    (继承)                       │
│                                          │
│  可通过 proxy-target-class="true" 强制CGLIB
└──────────────────────────────────────────┘
```

### 6.5 JDK 代理 vs CGLIB 代理 对比（细化）

| 对比项 | JDK 动态代理 | CGLIB |
|--------|------------|-------|
| 原理 | 实现目标接口，反射调用 | 继承目标类，字节码增强 |
| 要求 | 目标类必须有接口 | 目标类不能是 `final` |
| 性能 | 创建快，调用稍慢 | 创建稍慢，调用快（方法直接索引） |
| 无接口类 | 不支持 | 支持 |
| `final` 方法 | 不拦截 | 不拦截（无法 override） |
| Spring Boot 默认 | CGLIB（Spring Boot 2.x 起默认） | ✓ |

> **追问：Spring Boot 为什么默认改为 CGLIB？**  
> → JDK 代理要求注入时使用接口类型，如果注入的是具体类会抛出类型转换异常。CGLIB 更宽松，避免了这类问题，提升了开发体验。可通过 `spring.aop.proxy-target-class=false` 切回 JDK 代理。

### 6.6 AOP 源码创建入口

```

    ↓
postProcessAfterInitialization() 被每个 Bean 触发
    ↓
wrapIfNecessary() — 判断是否需要代理
    ↓
getAdvicesAndAdvisorsForBean() — 找匹配的 Advisors
    ↓
createProxy() — 创建代理对象（JDK / CGLIB）
```

## 七、Spring 事务

### 7.1 开启事务

```java
// 1. 注解方式
@EnableTransactionManagement
@Configuration
public class TransactionConfig {
    @Bean
    public DataSourceTransactionManager transactionManager(DataSource ds) {
        return new DataSourceTransactionManager(ds);
    }
}

// 2. 使用事务
@Service
public class UserService {
    @Transactional(isolation = Isolation.READ_COMMITTED, 
                   propagation = Propagation.REQUIRED)
    public void transferMoney() {
        // 业务逻辑
    }
}
```

### 7.2 传播行为

| 传播行为 | 说明 |
|----------|------|
| REQUIRED | 支持当前事务（默认） |
| REQUIRES_NEW | 新建事务，挂起当前 |
| SUPPORTS | 支持当前事务 |
| NOT_SUPPORTED | 非事务执行，挂起当前 |
| MANDATORY | 必须有事务，否则抛异常 |
| NEVER | 不能有事务 |
| NESTED | 嵌套事务（Savepoint） |

### 7.3 REQUIRED vs REQUIRES_NEW vs NESTED 对比（追问热点）

| 对比项 | REQUIRED | REQUIRES_NEW | NESTED |
|--------|---------|--------------|--------|
| 外层事务存在时 | 加入外层 | 挂起外层，新建独立事务 | 创建 Savepoint，嵌套在外层 |
| 内层回滚影响外层 | 是（同一事务） | 否（独立事务） | 否（回滚到 Savepoint） |
| 外层回滚影响内层 | 是 | 否 | 是（整体回滚） |
| 数据库支持要求 | 无 | 无 | 需要支持 Savepoint（如 MySQL InnoDB） |

### 7.4 事务原理

```
@EnableTransactionManagement
    ↓
注册 InfrastructureAdvisorAutoProxyCreator
    ↓
注册 TransactionInterceptor
    ↓
方法调用 → 代理对象 → TransactionInterceptor.invoke()
    ↓
TransactionTemplate.execute()
    ↓
DataSourceTransactionManager.getTransaction()
    ↓
获取连接 → 设置自动提交false → 执行SQL
    ↓
提交 / 回滚
```

### 7.5 事务与数据库连接绑定原理

```java
// TransactionSynchronizationManager — 事务核心工具类
// 使用 ThreadLocal 将 Connection 绑定到当前线程
private static final ThreadLocal<Map<Object, Object>> resources =
    new NamedThreadLocal<>("Transactional resources");

// 获取当前线程的连接
DataSourceUtils.getConnection(dataSource)
    → TransactionSynchronizationManager.getResource(dataSource)
    → 返回与当前事务绑定的 Connection
```

> **关键点**：同一线程内，所有通过 `DataSourceUtils` 获取连接的操作（包括 MyBatis、JdbcTemplate）都会复用同一个 `Connection`，从而保证在同一事务内。**跨线程则事务失效**，这是异步方法中事务不生效的根本原因。

### 8.2 核心组件

```java
// 1. DispatcherServlet - 前端控制器
public class DispatcherServlet extends HttpServlet {
    protected void doDispatch(HttpServletRequest request, 
                              HttpServletResponse response) {
        // 找到Handler
        HandlerExecutionChain mappedHandler = getHandler(request);
        // 获取适配器
        HandlerAdapter ha = getHandlerAdapter(mappedHandler.getHandler());
        // 执行
        mv = ha.handle(request, response, mappedHandler.getHandler());
        // 处理返回
        processDispatchResult(request, response, mappedHandler, mv);
    }
}
```

### 8.3 HandlerMapping 的多种实现（追问）

| 实现类 | 用途 |
|--------|------|
| `RequestMappingHandlerMapping` | 处理 `@RequestMapping` 注解，最常用 |
| `BeanNameUrlHandlerMapping` | 通过 BeanName 作为 URL 映射 |
| `RouterFunctionMapping` | Spring WebFlux 函数式路由 |
| `SimpleUrlHandlerMapping` | 静态资源映射 |

> `RequestMappingHandlerMapping` 在初始化时扫描所有 Bean，找到有 `@RequestMapping` 的方法，建立 URL → `HandlerMethod` 的映射关系，存储在 `MappingRegistry` 中。

### 8.4 拦截器执行顺序（细化）

```
HandlerInterceptor 的执行顺序（多个拦截器时）：

preHandle（顺序执行，返回false中断）
    ↓
Controller 方法执行
    ↓
postHandle（逆序执行，仅在Controller正常返回后）
    ↓
afterCompletion（逆序执行，无论是否异常都执行）
```

> **追问：`postHandle` 和 `afterCompletion` 的区别？**  
> → `postHandle` 在视图渲染**之前**，可以修改 `ModelAndView`；`afterCompletion` 在视图渲染**之后**，适合做资源清理（如释放连接、记录日志）。若 `preHandle` 返回 false，则 `postHandle` 不会执行，但已执行了 `preHandle` 的拦截器的 `afterCompletion` 仍会执行。

### 8.5 `@RequestBody` 参数解析原理

```
RequestMappingHandlerAdapter.invokeHandlerMethod()
    ↓
HandlerMethodArgumentResolverComposite.resolveArgument()
    ↓
匹配到 RequestResponseBodyMethodProcessor（处理 @RequestBody）
    ↓
HttpMessageConverter.read()
    ↓
MappingJackson2HttpMessageConverter → ObjectMapper.readValue()
    ↓
反序列化为 Java 对象
```

### 9.4 一级缓存

```java
// BaseExecutor - 一级缓存实现
public <E> List<E> query(MappedStatement ms, Object parameter, 
                         RowBounds rowBounds, ResultHandler handler) {
    // 1. 构建缓存Key
    CacheKey key = createCacheKey(ms, parameter, rowBounds, null);
    
    // 2. 查询缓存
    List<E> list = localCache.getObject(key);
    if (list != null) {
        return list;  // 缓存命中
    }
    
    // 3. 查数据库
    list = queryFromDatabase(ms, parameter, rowBounds, handler);
    
    // 4. 放入缓存
    localCache.putObject(key, list);
    return list;
}
```

特点：
- SqlSession 级别
- 默认开启
- 同一 SqlSession 多次查询相同语句命中

### 9.5 CacheKey 的构成（追问）

### 9.6 一级缓存失效场景

| 场景 | 原因 |
|------|------|
| 执行了 INSERT/UPDATE/DELETE | `clearLocalCache()` 被调用 |
| 手动调用 `sqlSession.clearCache()` | 显式清除 |
| 不同的 SqlSession | 各自独立的 `localCache` |
| `flushCache="true"` 的查询 | 每次查询前清空缓存 |
| `localCacheScope=STATEMENT` | 每条语句执行完即清空 |

### 9.7 Mapper 接口动态代理原理（追问高频）

> **追问：Mapper 接口没有实现类，MyBatis 是怎么执行的？**

```java
// MapperProxyFactory 创建代理
public T newInstance(SqlSession sqlSession) {
    MapperProxy<T> mapperProxy = new MapperProxy<>(sqlSession, mapperInterface, methodCache);
    return (T) Proxy.newProxyInstance(
        mapperInterface.getClassLoader(),
        new Class[]{mapperInterface},
        mapperProxy   // InvocationHandler
    );
}

// MapperProxy.invoke() — 实际执行逻辑
public Object invoke(Object proxy, Method method, Object[] args) {
    // Object 自身方法（toString/equals等）直接调用
    if (Object.class.equals(method.getDeclaringClass())) {
        return method.invoke(this, args);
    }
    // 从缓存获取 MapperMethod
    MapperMethod mapperMethod = cachedMapperMethod(method);
    // 根据 SQL 类型（SELECT/INSERT/UPDATE/DELETE）执行对应 SqlSession 方法
    return mapperMethod.execute(sqlSession, args);
}
```

流程总结：`getMapper()` → `MapperProxyFactory.newInstance()` → JDK 动态代理 → `MapperProxy.invoke()` → `SqlSession.selectOne/selectList/insert/...`
