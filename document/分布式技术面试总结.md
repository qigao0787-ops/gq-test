## 一、Redis

### 1.1 Redis 线程模型

```
┌─────────────────────────────────────────────────────┐
│            Redis 6.0 线程模型                        │
├─────────────────────────────────────────────────────┤
│                                                     │
│   主线程 (main)                                      │
│   ┌─────────────────────────────────────────────┐  │
│   │  1. 绑定端口                                  │  │
│   │  2. 接收连接                                  │  │
│   │  3. 读取命令                                  │  │
│   │  4. 解析命令                                  │  │
│   │  5. 执行命令                                  │  │
│   │  6. 返回结果                                  │  │
│   └─────────────────────────────────────────────┘  │
│                         │                           │
│            ┌────────────┼────────────┐             │
│            ▼            ▼            ▼             │
│      IO线程1      IO线程2      IO线程3             │
│      (网络读写)   (网络读写)   (网络读写)           │
│                                                     │
│   6.0之前：单线程（单 Reactor）                     │
│   6.0之后：多线程（主线程 + IO线程）                │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 1.2 数据结构

| 结构 | 底层实现 | 用途 |
|------|----------|------|
| String | SDS | 缓存、计数器、分布式锁 |
| List | QuickList | 消息队列、列表 |
| Hash | ziplist + hashtable | 购物车、对象存储 |
| Set | intset + hashtable | 点赞、标签 |
| ZSet | ziplist + skiplist | 排行榜、延迟队列 |

### 1.3 持久化

```java
// RDB - 定时快照
// 优点：恢复快，文件小
// 缺点：可能丢失数据

// AOF - 追加日志
// 优点：数据完整
// 缺点：文件大，恢复慢
// appendfsync: always/everysec/no
```

### 1.4 过期策略

```java
// 1. 定期删除 - 随机抽查过期key
// 2. 惰性删除 - 访问时检查
// 3. 内存淘汰策略
//    - noeviction: 拒绝写入
//    - allkeys-lru: 最近最少使用
//    - volatile-lru: 带过期key的LRU
//    - allkeys-random: 随机
//    - volatile-ttl: 过期时间最短
```

### 1.5 分布式锁

```java
// 1. SET lock_key unique_value NX PX 30000
// 2. 释放锁：Lua脚本保证原子性
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end

// Redisson 实现
RLock lock = redisson.getLock("myLock");
lock.lock();
try {
    // 业务逻辑
} finally {
    lock.unlock();
}
```

## 二、MySQL 读写分离 & 分库分表

### 2.1 ShardingSphere 原理

```java
// 逻辑表
sql: SELECT * FROM t_order WHERE user_id = ?

// 解析 → 改写 → 路由 → 执行 → 归并
// 改写: t_order -> t_order_0, t_order_1
// 路由: 根据 user_id % 2 决定库
// 归并: 排序/分页/聚合
```

### 2.2 分片策略

| 策略 | 说明 |
|------|------|
| 哈希分片 | mod(hash(key)) |
| 范围分片 | BETWEEN AND |
| 亲和性分片 | 就近路由 |

### 2.3 分页问题

```java
// 错误：每个库查limit 10，然后合并排序取前10
// 正确：先查每个库前N页，再归并

// 方案：二次分页
// 1. 查每个库的 (page-1)*size + size 条
// 2. 内存中排序分页
```

## 三、Kafka

### 3.1 架构

```
┌─────────────────────────────────────────────────────┐
│               Kafka 架构                             │
├─────────────────────────────────────────────────────┤
│                                                     │
│   Producer                                          │
│       │                                            │
│       ▼                                            │
│   Broker (Leader/Follower)                         │
│       │                                            │
│   Topic ──► Partition ──► Replica                  │
│                │              │                     │
│                ▼              ▼                     │
│            Leader          Follower                 │
│                                                     │
│   Consumer Group                                    │
│       │                                            │
│       ▼                                            │
│   Partition (每个分区只被一个消费者消费)              │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 3.2 顺序保证

```java
// 1. 单分区内有序
// 2. 全局有序：单分区 + 单消费者
// 3. 消息key相同进入同一分区
producer.send(new ProducerRecord<>("topic", key, value));
```

### 3.3 消费模式

```java
// 1. 拉取模式
consumer.poll(Duration.ofMillis(100));

// 2. 消费者偏移量管理
// 自动提交：enable.auto.commit=true
// 手动提交：consumer.commitSync();
```

### 3.4 高性能原因

```
1. 顺序写磁盘 - 追加写
2. 零拷贝 - sendfile系统调用
3. 批量处理 - 压缩消息
4. 分区并行 - 水平扩展
5. 稀疏索引 - 高效查找
```

## 四、RabbitMQ

### 4.1 架构

```
Producer → Exchange → Binding → Queue → Consumer
```

### 4.2 消息确认

```java
// 1. 手动ACK
channel.basicConsume("queue", false, 
    (tag, msg) -> {
        try {
            // 处理消息
            channel.basicAck(tag, false);  // 确认
        } catch (Exception e) {
            channel.basicNack(tag, false, true);  // 拒绝
        }
    });

// 2. 消息属性
// persistent: 持久化
// deliveryMode: 2
```

NioEventLoopGroup workerGroup = new NioEventLoopGroup();

// 2. ServerBootstrap - 启动器
ServerBootstrap b = new ServerBootstrap();
b.group(bossGroup, workerGroup)
 .channel(NioServerSocketChannel.class)
 .childHandler(new ChannelInitializer<SocketChannel>() {
     @Override
     protected void initChannel(SocketChannel ch) {
         ch.pipeline().addLast(new HttpServerCodec());
         ch.pipeline().addLast(new SimpleChannelInboundHandler());
     }
 });

// 3. ChannelPipeline - 处理器链
