## 一、AI Agent 基础概念

### 1.1 什么是 AI Agent

```
┌─────────────────────────────────────────────────────┐
│              AI Agent 核心定义                       │
├─────────────────────────────────────────────────────┤
│                                                     │
│  AI Agent = LLM + 记忆 + 工具 + 规划能力            │
│                                                     │
│  特征:                                             │
│  - 自主性：能独立决策和行动                          │
│  - 反应性：能感知环境变化并响应                      │
│  - 主动性：能主动规划并执行目标                      │
│  - 社交能力：能与其他 Agent 或人类协作               │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 1.2 Agent vs Chatbot vs Copilot

| 维度 | Chatbot | Copilot | Agent |
|------|---------|---------|-------|
| 交互模式 | 问答式 | 辅助式 | 自主式 |
| 决策能力 | 无 | 建议 | 独立决策+执行 |
| 工具使用 | 无/有限 | 有限 | 丰富 |
| 记忆 | 短期对话 | 上下文 | 长短期记忆 |
| 典型产品 | ChatGPT | GitHub Copilot | Claude Code / Devin |

## 二、核心设计模式

### 2.1 ReAct (Reasoning + Acting)

```
┌─────────────────────────────────────────────────────┐
│              ReAct 循环                              │
├─────────────────────────────────────────────────────┤
│                                                     │
│  用户问题                                           │
│      │                                             │
│      ▼                                             │
│  ┌──────────┐                                      │
│  │ Thought  │  推理：分析当前情况                    │
│  └────┬─────┘                                      │
│       ▼                                            │
│  ┌──────────┐                                      │
│  │  Action  │  行动：调用工具/API                   │
│  └────┬─────┘                                      │
│       ▼                                            │
│  ┌──────────┐                                      │
│  │Observation│  观察：获取工具返回结果               │
│  └────┬─────┘                                      │
│       │                                            │
│       ▼ 循环直到得出最终答案                         │
│  ┌──────────┐                                      │
│  │  Answer  │  最终回答                             │
│  └──────────┘                                      │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 2.2 Plan-and-Execute

```
用户目标
    │
    ▼
┌──────────┐
│  Planner │  生成步骤列表
└────┬─────┘
     │
     ▼
Step 1 → Execute → Result
Step 2 → Execute → Result
Step 3 → Execute → Result
     │
     ▼
┌──────────┐
│ Replan?  │  根据中间结果决定是否调整计划
└────┬─────┘
     │
     ▼
最终结果
```

### 2.3 常见 Agent 设计模式对比

| 模式 | 原理 | 优点 | 缺点 | 代表框架 |
|------|------|------|------|----------|
| ReAct | 思考→行动→观察循环 | 简单直觉 | 长任务易迷失 | LangChain |
| Plan-and-Execute | 先规划再执行 | 结构清晰 | 规划可能不准 | LangGraph |
| Reflexion | 执行后自我反思 | 能自我纠错 | 耗费更多 token | Reflexion |
| Tree of Thoughts | 多路径探索 | 复杂推理强 | 计算成本高 | ToT |
| Multi-Agent | 多 Agent 协作 | 专业分工 | 协调复杂 | CrewAI/AutoGen |

## 三、Function Calling 与工具使用

### 3.1 Function Calling 机制

```json
// 定义工具 schema
{
  "name": "search_web",
  "description": "搜索互联网获取最新信息",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "搜索关键词"
      }
    },
    "required": ["query"]
  }
}

// LLM 返回 tool_call
{
  "tool_calls": [{
    "id": "call_123",
    "function": {
      "name": "search_web",
      "arguments": "{\"query\": \"2024 AI trends\"}"
    }
  }]
}
```

### 3.2 工具设计原则

| 原则 | 说明 |
|------|------|
| 单一职责 | 每个工具只做一件事 |
| 清晰描述 | description 决定 LLM 能否正确选择工具 |
| 参数明确 | 用 JSON Schema 严格定义参数类型和约束 |
| 错误处理 | 工具返回结构化错误信息，Agent 可重试 |
| 幂等性 | 同一调用多次结果一致，避免副作用 |

### 3.3 MCP (Model Context Protocol)

```
┌─────────────────────────────────────────────────────┐
│         MCP 协议架构                                │
├─────────────────────────────────────────────────────┤
│                                                     │
│  Host (IDE/App)                                     │
│      │                                             │
│  MCP Client                                        │
│      │  (JSON-RPC over stdio/SSE)                  │
│      ▼                                             │
│  MCP Server 1  ─── Tools (工具)                    │
│  MCP Server 2  ─── Resources (资源)               │
│  MCP Server 3  ─── Prompts (提示模板)              │
│                                                     │
│  核心能力:                                          │
│  - Tools: 可执行的函数                              │
│  - Resources: 可读取的数据源                        │
│  - Prompts: 预定义的提示模板                        │
│  - Sampling: 服务端请求 LLM 补全                    │
│                                                     │
└─────────────────────────────────────────────────────┘
```

## 四、Prompt Engineering 进阶

### 4.1 System Prompt 设计

```markdown
# Agent System Prompt 关键要素

1. 角色定义 (Role)
   - 你是一个专业的 XX 助手

2. 能力边界 (Capabilities)
   - 你能做什么，不能做什么

3. 工具说明 (Tools)
   - 可用工具列表和使用时机

4. 行为规范 (Guidelines)
   - 输出格式、安全约束

5. 示例 (Examples)
   - Few-shot 示例引导行为
```

### 4.2 常用 Prompt 技巧

| 技巧 | 说明 | 场景 |
|------|------|------|
| Chain-of-Thought | 让模型逐步推理 | 复杂逻辑推理 |
| Few-shot | 给出示例 | 格式规范化输出 |
| Self-Consistency | 多次生成取多数 | 提高答案准确性 |
| Structured Output | 强制 JSON/XML 输出 | 工具调用、数据提取 |
| Role Playing | 角色扮演 | 领域专家模拟 |

## 五、RAG (检索增强生成)

### 5.1 RAG 流程

```
┌─────────────────────────────────────────────────────┐
│              RAG 管线                                │
├─────────────────────────────────────────────────────┤
│                                                     │
│  离线阶段 (Indexing):                               │
│  文档 → 分块(Chunking) → Embedding → 向量数据库     │
│                                                     │
│  在线阶段 (Retrieval + Generation):                 │
│  Query → Embedding → 向量检索 Top-K                 │
│       → 检索结果 + Query → LLM → 回答              │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 5.2 分块策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 固定大小 | 按字符/token 数分割 | 通用 |
| 递归分割 | 按段落→句子→字符递归 | 结构化文档 |
| 语义分割 | 按语义相似度切分 | 长文档 |
| 文档结构 | 按 Markdown 标题分割 | 技术文档 |

### 5.3 向量数据库对比

| 数据库 | 特点 | 适用场景 |
|--------|------|----------|
| Pinecone | 全托管、易用 | 生产环境快速接入 |
| Milvus | 开源、高性能 | 大规模生产部署 |
| ChromaDB | 轻量级、Python 友好 | 原型开发 |
| Weaviate | GraphQL API、混合搜索 | 复杂查询 |
| pgvector | PostgreSQL 扩展 | 已有 PG 基础设施 |

### 5.4 RAG 优化手段

| 方向 | 技术 | 效果 |
|------|------|------|
| 检索优化 | Hybrid Search (向量+关键词) | 提高召回率 |
| 重排序 | Cross-encoder Reranker | 提高精度 |
| Query 改写 | HyDE / Multi-Query | 提高匹配度 |
| 上下文压缩 | 去除无关内容 | 减少噪音 |
| Chunk 优化 | Parent-Child / 重叠窗口 | 保留上下文 |

## 六、主流 Agent 框架

### 6.1 LangChain / LangGraph

```python
# LangGraph 基本结构
from langgraph.graph import StateGraph, END

# 定义状态
class AgentState(TypedDict):
    messages: list
    next_action: str

# 构建图
graph = StateGraph(AgentState)
graph.add_node("agent", agent_node)
graph.add_node("tools", tool_node)
graph.add_edge("agent", "tools")
graph.add_conditional_edges("tools", should_continue)

app = graph.compile()
```

### 6.2 CrewAI (多 Agent 协作)

```python
from crewai import Agent, Task, Crew

researcher = Agent(
    role="研究员",
    goal="搜集最新信息",
    tools=[search_tool]
)

writer = Agent(
    role="作者",
    goal="将研究结果整理成文章",
    tools=[write_tool]
)

crew = Crew(
    agents=[researcher, writer],
    tasks=[research_task, write_task],
    process=Process.sequential  # 顺序执行
)
```

### 6.3 框架选型

| 框架 | 定位 | 优点 | 缺点 |
|------|------|------|------|
| LangChain | 通用 LLM 应用框架 | 生态丰富、组件多 | 抽象重、学习曲线 |
| LangGraph | 有状态 Agent 编排 | 可控性强、可视化 | 概念多 |
| CrewAI | 多 Agent 协作 | 简单易用 | 灵活性不足 |
| AutoGen | 多 Agent 对话 | 微软出品、灵活 | 复杂场景难调试 |
| Dify | 低代码 Agent 平台 | 可视化编排 | 自定义受限 |
| Coze | 字节出品 Agent 平台 | 易用、插件丰富 | 生态封闭 |

## 七、记忆系统

### 7.1 记忆类型

```
┌─────────────────────────────────────────────────────┐
│              Agent 记忆体系                          │
├─────────────────────────────────────────────────────┤
│                                                     │
│  短期记忆 (Short-term Memory)                       │
│  - 当前对话上下文 (Context Window)                  │
│  - 工作记忆 (当前任务中间状态)                      │
│                                                     │
│  长期记忆 (Long-term Memory)                        │
│  - 向量存储 (语义检索历史信息)                      │
│  - 知识图谱 (结构化关系存储)                        │
│  - Key-Value 存储 (用户偏好/事实)                   │
│                                                     │
│  外部记忆 (External Memory)                         │
│  - 文件系统 / 数据库                               │
│  - 搜索引擎索引                                    │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 7.2 上下文窗口管理

| 策略 | 说明 | 优缺点 |
|------|------|--------|
| 滑动窗口 | 只保留最近 N 轮 | 简单，丢失早期信息 |
| 摘要压缩 | 用 LLM 总结历史 | 保留关键信息，有信息损失 |
| RAG 记忆 | 历史存向量库按需检索 | 灵活，检索可能不准 |
| 混合策略 | 最近几轮原文 + 历史摘要 | 平衡性好 |

## 九、评估与可观测性

### 9.1 Agent 评估维度

| 维度 | 指标 | 说明 |
|------|------|------|
| 任务完成率 | Success Rate | 最终目标是否达成 |

### 9.2 可观测性工具

| 工具 | 功能 | 特点 |
|------|------|------|
| LangSmith | Trace / 评估 / 监控 | LangChain 生态 |

### 9.3 常见问题排查

| 问题 | 可能原因 | 解决 |
|------|---------|------|
| 工具调用失败 | Schema 描述不清 | 优化 description |
| 无限循环 | 缺少终止条件 | 设置 max_iterations |
| 幻觉输出 | 缺乏 grounding | 增加 RAG / 工具验证 |
| 上下文溢出 | 对话太长 | 摘要压缩 / 分段处理 |
| 响应慢 | 串行调用太多 | 并行化 / 缓存 |

## 十、安全与部署

### 10.1 安全考量

| 风险 | 描述 | 对策 |
|------|------|------|
| Prompt 注入 | 用户通过输入覆盖指令 | 输入清洗 + 权限隔离 |
| 工具滥用 | Agent 执行危险操作 | 白名单 + 人工审批 |
| 数据泄露 | 敏感信息通过 LLM 泄露 | 数据脱敏 + 权限控制 |
| 成本失控 | 无限循环消耗 token | 设置 budget / 超时 |
| 幻觉 | 生成不准确信息 | RAG + 事实校验 |
