## 二、TCP 协议

### 2.1 三次握手

```
┌──────────┐                      ┌──────────┐
│  Client  │                      │  Server  │
└────┬─────┘                      └────┬─────┘
     │                                 │
     │  ① SYN=1, seq=x                │
     │────────────────────────────────→│  Server: LISTEN → SYN_RCVD
     │                                 │
     │  ② SYN=1, ACK=1, seq=y, ack=x+1│
     │←────────────────────────────────│
     │                                 │
     │  ③ ACK=1, seq=x+1, ack=y+1     │
     │────────────────────────────────→│  ESTABLISHED
     │                                 │
     │         连接建立,可以传数据       │
```

**为什么三次？**
- 两次: 服务端无法确认客户端收到了 SYN+ACK（可能是历史连接）
- 三次: 双方都能确认对方的收发能力正常
- 防止已失效的连接请求到达服务端导致资源浪费

### 2.2 四次挥手

```
┌──────────┐                      ┌──────────┐
│  Client  │                      │  Server  │
└────┬─────┘                      └────┬─────┘
     │                                 │
     │  ① FIN=1, seq=u                 │
     │────────────────────────────────→│  Server: CLOSE_WAIT
     │                                 │
     │  ② ACK=1, ack=u+1              │
     │←────────────────────────────────│  (服务端可能还有数据要发)
     │                                 │
     │  ③ FIN=1, seq=w                 │
     │←────────────────────────────────│  Server: LAST_ACK
     │                                 │
     │  ④ ACK=1, ack=w+1              │
     │────────────────────────────────→│  Server: CLOSED
     │                                 │
     │  TIME_WAIT (2MSL)               │
     │  ...然后 CLOSED                  │
```

**为什么四次？**
- TCP 是全双工，每个方向需要单独关闭
- 服务端收到 FIN 后可能还有数据未发完，先 ACK 再发自己的 FIN

**为什么 TIME_WAIT 等 2MSL？**
- 确保最后一个 ACK 能到达（丢了对方会重发 FIN）
- 让本连接的所有报文在网络中消失，不影响新连接

### 2.3 TCP 可靠传输机制

| 机制 | 说明 |
|------|------|
| 序列号/确认号 | 每个字节编号，ACK 确认收到 |
| 超时重传 | RTO 内未收到 ACK 则重传 |
| 滑动窗口 | 控制发送速率，不必逐个等 ACK |
| 流量控制 | 接收方通过窗口大小告知发送方 |
| 拥塞控制 | 慢启动→拥塞避免→快重传→快恢复 |
| 校验和 | 检测传输错误 |

### 2.4 TCP 拥塞控制

```
窗口大小
    │        拥塞避免(线性增长)
    │       ╱
    │      ╱  ← ssthresh
    │     ╱
    │    ╱ 
    │   ╱ 慢启动(指数增长)
    │  ╱
    │ ╱
    │╱
    └─────────────────────────── 时间

四个阶段:
1. 慢启动: cwnd 从1开始指数增长
2. 拥塞避免: cwnd ≥ ssthresh 后线性增长
3. 快重传: 收到3个重复ACK立即重传(不等超时)
4. 快恢复: ssthresh = cwnd/2, cwnd = ssthresh + 3
```

### 2.5 TCP vs UDP

| 维度 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接(三次握手) | 无连接 |
| 可靠性 | 可靠(重传/排序/去重) | 不可靠 |
| 顺序 | 保证有序 | 不保证 |
| 速度 | 较慢(确认机制) | 快 |
| 头部 | 20字节 | 8字节 |
| 场景 | HTTP/文件传输/邮件 | DNS/视频/游戏/直播 |

## 三、HTTP 协议

### 3.1 HTTP 请求/响应格式

```
请求:
GET /api/users HTTP/1.1        ← 请求行
Host: example.com              ← 请求头
Content-Type: application/json
                               ← 空行
{request body}                 ← 请求体

响应:
HTTP/1.1 200 OK                ← 状态行
Content-Type: application/json ← 响应头
Content-Length: 128
                               ← 空行
{response body}                ← 响应体
```

### 3.2 HTTP 状态码

| 分类 | 含义 | 常见 |
|------|------|------|
| 1xx | 信息 | 100 Continue |
| 2xx | 成功 | 200 OK / 201 Created / 204 No Content |
| 3xx | 重定向 | 301 永久 / 302 临时 / 304 未修改(缓存) |
| 4xx | 客户端错误 | 400 Bad Request / 401 未认证 / 403 禁止 / 404 未找到 |
| 5xx | 服务端错误 | 500 内部错误 / 502 Bad Gateway / 503 服务不可用 |

### 3.3 HTTP 方法

| 方法 | 语义 | 幂等 | 安全 |
|------|------|:----:|:----:|
| GET | 获取资源 | ✅ | ✅ |
| POST | 创建资源 | ❌ | ❌ |
| PUT | 全量更新 | ✅ | ❌ |
| PATCH | 部分更新 | ❌ | ❌ |
| DELETE | 删除资源 | ✅ | ❌ |
| HEAD | 获取头信息 | ✅ | ✅ |
| OPTIONS | 查询支持方法(CORS预检) | ✅ | ✅ |

### 3.4 HTTP 1.0 / 1.1 / 2.0 / 3.0

| 版本 | 关键特性 |
|------|---------|
| 1.0 | 短连接(每次请求新建TCP) |
| 1.1 | 长连接(Connection: keep-alive) / 管线化 / Host头 / 分块传输 |
| 2.0 | 多路复用 / 头部压缩(HPACK) / 二进制帧 / 服务端推送 |
| 3.0 | 基于 QUIC(UDP) / 0-RTT 连接 / 无队头阻塞 |

**HTTP/2 多路复用：**
```
HTTP/1.1: 
  请求1 ─────────→ 响应1 ─────→ 请求2 ─────→ 响应2 (串行/队头阻塞)

HTTP/2:
  请求1 ─┐                   ┌─ 响应1
  请求2 ─┼── 同一TCP连接 ────┼─ 响应2  (并行,无阻塞)
  请求3 ─┘                   └─ 响应3
```

## 四、HTTPS

### 4.1 HTTPS = HTTP + TLS/SSL

```
┌─────────────────────────────────────────────────────┐
│              HTTPS 握手过程 (TLS 1.2)               │
├─────────────────────────────────────────────────────┤
│                                                     │
│  Client                          Server             │
│    │                               │                │
│    │  ① ClientHello               │                │
│    │  (支持的加密套件/随机数A)      │                │
│    │──────────────────────────────→│                │
│    │                               │                │
│    │  ② ServerHello               │                │
│    │  (选定加密套件/随机数B/证书)   │                │
│    │←──────────────────────────────│                │
│    │                               │                │
│    │  ③ 验证证书(CA链)             │                │
│    │  生成预主密钥(Pre-Master)      │                │
│    │  用服务端公钥加密发送          │                │
│    │──────────────────────────────→│                │
│    │                               │                │
│    │  ④ 双方用 随机数A+B+Pre-Master │                │
│    │  生成对称密钥(Session Key)     │                │
│    │                               │                │
│    │  ⑤ 后续通信用对称密钥加密      │                │
│    │←─────────────────────────────→│                │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 4.2 对称加密 vs 非对称加密

| 维度 | 对称加密 | 非对称加密 |
|------|---------|-----------|
| 密钥 | 同一把 | 公钥+私钥 |
| 速度 | 快 | 慢(100~1000倍) |
| 算法 | AES / DES | RSA / ECC |
| HTTPS用途 | 传输数据 | 交换对称密钥 |

### 4.3 证书验证流程

```
1. 服务端发送证书(含公钥+域名+CA签名)
2. 客户端用 CA 公钥验证签名(浏览器内置根CA)
3. 验证域名是否匹配
4. 验证有效期
5. 通过 → 信任该公钥
```

## 五、DNS

### 5.1 DNS 解析流程

```
浏览器缓存 → 系统缓存(hosts) → 本地DNS服务器
    → 根DNS → 顶级DNS(.com) → 权威DNS
    → 返回 IP 地址

查询方式:
- 递归查询: 客户端→本地DNS (本地DNS帮你查到底)
- 迭代查询: 本地DNS→根→顶级→权威 (每次返回下一步地址)
```

## 六、WebSocket

### 6.1 WebSocket vs HTTP

| 维度 | HTTP | WebSocket |
|------|------|-----------|
| 连接 | 短连接/长连接(半双工) | 全双工持久连接 |
| 方向 | 客户端发起 | 双方都可主动发 |
| 头部 | 每次携带完整头 | 握手后极小帧头(2-14字节) |
| 场景 | 请求-响应 | 实时通信/推送/聊天 |

### 6.2 WebSocket 握手

```
客户端:
GET /chat HTTP/1.1
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==

服务端:
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

→ 后续使用 WebSocket 帧通信
```

## 七、Cookie / Session / Token

| 维度 | Cookie | Session | JWT Token |
|------|--------|---------|-----------|
| 存储 | 客户端浏览器 | 服务端内存/Redis | 客户端 |
| 大小 | ≤4KB | 无限制 | 较大 |
| 安全 | 可被窃取 | 相对安全 | 签名防篡改 |
| 跨域 | 不支持 | 不支持 | 支持(Header) |
| 分布式 | 天然支持 | 需共享(Redis) | 天然支持 |
| 有效期 | 可设置 | 服务端控制 | Token内含过期时间 |

## 八、网络安全

### 8.1 常见攻击

| 攻击 | 原理 | 防御 |
|------|------|------|
| XSS | 注入恶意脚本 | 转义输出/CSP/HttpOnly Cookie |
| CSRF | 伪造用户请求 | CSRF Token/SameSite Cookie/Referer检查 |
| SQL注入 | 拼接恶意SQL | PreparedStatement/参数化查询 |
| DDoS | 大量请求耗尽资源 | CDN/限流/黑名单/高防IP |
| 中间人 | 劫持通信 | HTTPS/证书固定 |
