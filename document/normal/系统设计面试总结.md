## 二、秒杀系统 ★★★★★

### 2.1 核心挑战

| 挑战 | 说明 |
|------|------|
| 瞬时高并发 | 百万级 QPS 涌入 |
| 超卖 | 库存扣减必须准确 |
| 恶意请求 | 刷子/机器人 |
| 系统雪崩 | 下游撑不住 |

### 2.2 架构设计

```
┌─────────────────────────────────────────────────────┐
│                  秒杀系统架构                        │
├─────────────────────────────────────────────────────┤
│                                                     │
│  用户 → CDN(静态页) → Nginx(限流/过滤)              │
│         → 网关(风控/令牌桶)                         │
│         → 秒杀服务(Redis预扣库存)                   │
│         → MQ 异步下单                              │
│         → 订单服务 → DB                            │
│                                                     │
│  核心思路: 层层过滤,漏斗模型                        │
│  100万请求 → 10万到网关 → 1万到服务 → 100到DB      │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 2.3 关键方案

| 问题 | 方案 |
|------|------|
| 前端削峰 | 按钮置灰/验证码/答题/随机延迟 |
| Nginx 限流 | `limit_req_zone` 令牌桶 |
| 库存预热 | 活动前把库存加载到 Redis |
| 扣减库存 | Redis DECR 原子操作(Lua脚本防超卖) |
| 异步下单 | 扣库存成功 → 发 MQ → 消费者创建订单 |
| 兜底降级 | Sentinel 熔断 / 返回排队中 |

### 2.4 Redis Lua 防超卖

```lua
-- 原子扣减库存
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock > 0 then
    redis.call('DECR', KEYS[1])
    return 1  -- 成功
else
    return 0  -- 库存不足
end
```

## 三、短链接系统 ★★★★

### 3.1 核心需求

- 长URL → 短码 (如 `https://t.cn/A6x8kP`)
- 短码 → 重定向到长URL
- 高并发读,写少

### 3.2 方案设计

```
┌─────────────────────────────────────────────────────┐
│              短链接架构                              │
├─────────────────────────────────────────────────────┤
│                                                     │
│  写入流程:                                          │
│  长URL → Hash/发号器 → Base62编码 → 存DB+Redis      │
│                                                     │
│  读取流程:                                          │
│  短码 → Redis 查缓存 → 未命中查 DB → 302 重定向     │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 3.3 短码生成方案

| 方案 | 原理 | 优缺点 |
|------|------|--------|
| Hash(MD5/MurmurHash) | 取前6位 | 简单,有冲突 |
| 自增ID + Base62 | ID=12345→Base62 | 无冲突,可预测 |
| Snowflake + Base62 | 分布式ID | 无冲突,不连续 |
| 预生成号段 | 提前生成短码池 | 性能好,需维护 |

```java
// Base62 编码 (0-9 + a-z + A-Z = 62个字符)
// 6位Base62 = 62^6 ≈ 568亿种组合
private static final String BASE62 = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ";

public String encode(long id) {
    StringBuilder sb = new StringBuilder();
    while (id > 0) {
        sb.append(BASE62.charAt((int)(id % 62)));
        id /= 62;
    }
    return sb.reverse().toString();
}
```

### 3.4 301 vs 302

| 状态码 | 含义 | 短链选择 |
|--------|------|---------|
| 301 | 永久重定向(浏览器缓存) | 无法统计点击量 |
| 302 | 临时重定向(每次请求服务) | ✅ 可以统计/分析 |

## 四、分布式ID ★★★★

### 4.1 需求

- 全局唯一
- 趋势递增(B+Tree 友好)
- 高性能(不能成为瓶颈)
- 高可用

### 4.2 方案对比

| 方案 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| UUID | 128bit随机 | 简单,无依赖 | 无序,索引差,太长 |
| 数据库自增 | auto_increment | 简单有序 | 单点瓶颈 |
| 数据库号段 | 每次取一段(如1000~1999) | 减少DB访问 | 需要额外服务 |
| Redis INCR | 原子自增 | 高性能 | 依赖Redis,持久化问题 |
| Snowflake | 时间+机器+序列号 | 有序,高性能,无依赖 | 时钟回拨 |
| Leaf (美团) | 号段+Snowflake 双模式 | 生产验证 | 复杂 |

### 4.3 Snowflake 结构

```
┌─┬───────────────────────────────────────┬──────────┬──────────┐
│0│  41 bit 时间戳(ms)                     │10bit机器 │12bit序号 │
└─┴───────────────────────────────────────┴──────────┴──────────┘
  │                                         │          │
  符号位                                  1024节点   4096/ms

- 支持 69 年
- 每毫秒每节点 4096 个 ID
- 问题: 时钟回拨 → 等待/报错/Leaf改进
```

## 五、限流方案 ★★★★

### 5.1 四种限流算法

| 算法 | 原理 | 特点 |
|------|------|------|
| 计数器 | 固定窗口计数 | 简单,有临界突刺问题 |
| 滑动窗口 | 细分时间格子 | 解决临界问题 |
| 漏桶 | 匀速流出 | 强制匀速,无法应对突发 |
| 令牌桶 | 匀速放令牌,桶满丢弃 | ✅ 允许一定突发流量 |

### 5.2 令牌桶算法

```
      ┌─────────┐
      │ 令牌生产 │  固定速率放令牌
      └────┬────┘
           ▼
      ┌─────────┐
      │  令牌桶  │  容量有上限(突发量)
      └────┬────┘
           ▼
      请求到来 → 有令牌? → 通过
                    ↓ 无令牌
                  拒绝/排队
```

### 5.3 实现方案

| 层级 | 方案 | 工具 |
|------|------|------|
| 网关层 | Nginx limit_req | 令牌桶 |
| 应用层 | Guava RateLimiter | 单机令牌桶 |
| 分布式 | Redis + Lua | 分布式令牌桶 |
| 框架 | Sentinel / Resilience4j | 规则灵活 |

```java
// Guava RateLimiter
RateLimiter limiter = RateLimiter.create(100);  // 100 QPS
if (limiter.tryAcquire()) {
    // 处理请求
} else {
    // 限流,返回 429
}
```

## 六、Feed 流系统 ★★★

### 6.1 场景

微博/朋友圈/抖音: 用户发布内容,关注者能看到

### 6.2 推拉模型

| 模型 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 拉模型(读扩散) | 读时聚合关注人动态 | 写简单,无冗余 | 读慢,大V关注多 |
| 推模型(写扩散) | 写时推送到粉丝收件箱 | 读快 | 大V写放大(百万粉丝) |
| 推拉结合 | 大V用拉,普通人用推 | ✅ 平衡 | 实现复杂 |

### 6.3 推拉结合方案

```
普通用户发布 → 推送到所有粉丝的 inbox (Redis ZSet, score=时间戳)
大V发布 → 只写 outbox, 粉丝读时合并

读取 Feed:
1. 从 inbox (Redis ZSet) 拉自己的推送
2. 拉关注的大V的 outbox
3. 合并排序,分页返回
```

## 七、延迟任务 ★★★

### 7.1 方案对比

| 方案 | 精度 | 可靠性 | 适用 |
|------|------|--------|------|
| JDK DelayQueue | 毫秒 | 低(单机/内存) | 单机小量 |
| ScheduledThreadPool | 毫秒 | 低 | 定时任务 |
| Redis ZSet | 秒 | 中 | 分布式延迟 |
| RocketMQ 延迟消息 | 固定级别 | 高 | 18个级别 |
| 时间轮(Netty/Kafka) | 毫秒 | 中 | 高性能 |
| XXL-Job | 秒 | 高 | 分布式调度 |

### 7.2 Redis ZSet 实现延迟队列

```java
// 写入: score = 执行时间戳
redis.zadd("delay_queue", System.currentTimeMillis() + 30*60*1000, orderId);

// 消费: 轮询取到期任务
Set<String> tasks = redis.zrangeByScore("delay_queue", 0, System.currentTimeMillis());
for (String task : tasks) {
    if (redis.zrem("delay_queue", task) > 0) {  // 原子移除防重复消费
        process(task);
    }
}
```

## 八、其他常见设计题

### 8.1 分布式锁

| 方案 | 实现 | 特点 |
|------|------|------|

### 8.2 分布式事务

| 方案 | 一致性 | 性能 | 场景 |
|------|--------|------|------|
| 2PC/XA | 强一致 | 差 | 金融 |
| TCC | 强一致 | 中 | 金融(Try-Confirm-Cancel) |
| SAGA | 最终一致 | 好 | 长事务+补偿 |
| 本地消息表 | 最终一致 | 好 | 通用 |
| RocketMQ事务消息 | 最终一致 | 好 | ✅ 推荐 |
| Seata AT | 强一致 | 中 | ✅ 低侵入 |

### 8.3 接口幂等性

| 方案 | 实现 |
|------|------|
| Token 机制 | 请求前获取token,请求时携带,服务端校验后删除 |

| 状态机 | 只允许状态正向流转 |
