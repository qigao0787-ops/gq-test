> 整理自：线上故障排查 + 项目经验包装方法论
> 适用：Java 后端面试（中高级岗必考项目深度）

## 一、STAR 答题法

### 1.1 STAR 模板

```
S - Situation (背景): 什么项目/什么业务场景
T - Task (任务): 你负责什么/要解决什么问题
A - Action (行动): 你具体做了什么(技术方案)
R - Result (结果): 带来了什么收益(量化数据)
```

### 1.2 示例：缓存优化

```
S: 电商平台商品详情页，大促期间 QPS 从 5000 飙到 5 万，DB 频繁超时
T: 我负责优化商品详情的查询性能，目标 RT < 50ms
A: 
  1. 引入 Redis 缓存，热门商品预热
  2. 使用本地缓存(Caffeine)做二级缓存，减少 Redis 网络开销
  3. 缓存击穿用互斥锁 + 逻辑过期方案
  4. 缓存雪崩用随机 TTL + 集群部署
R: 
  - RT 从 200ms 降到 30ms
  - DB QPS 降低 95%
  - 大促期间零超时
```

### 1.3 包装原则

| 原则 | 说明 |
|------|------|
| 量化结果 | 用数字说话：QPS/RT/可用性/成本 |
| 突出难点 | 说清楚"为什么难"、你怎么解决的 |
| 技术深度 | 提到你用的技术的底层原理 |
| 对比选型 | 为什么选A不选B |
| 复盘改进 | 遇到什么坑，怎么改进的 |

## 二、线上故障排查 ★★★★★

### 2.1 CPU 100% 排查

```bash
# 1. 找到 Java 进程
top -c                          # 找 CPU 高的 PID

# 2. 找到最耗 CPU 的线程
top -Hp <pid>                   # 找线程 TID

# 3. 线程 ID 转十六进制
printf "%x\n" <tid>             # 如 12345 → 3039

# 4. jstack 查看线程堆栈
jstack <pid> | grep "3039" -A 30

# 5. 或者直接用 Arthas
arthas> thread -n 3             # 最耗 CPU 的 3 个线程
```

**常见原因：**
| 原因 | 特征 | 解决 |
|------|------|------|
| 死循环 | 同一线程一直在某方法 | 修复逻辑 |
| 频繁 GC | GC 线程占满 CPU | 查内存泄漏 |
| 正则回溯 | 正则匹配耗 CPU | 优化正则表达式 |
| 加密/序列化 | 计算密集 | 缓存结果/异步 |

### 2.2 内存溢出(OOM)排查

```bash
# 1. 确保开启了 OOM dump
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/tmp/heap.hprof

# 2. 获取 dump (在线)
jmap -dump:format=b,file=heap.hprof <pid>
# 或 Arthas: heapdump /tmp/heap.hprof

# 3. MAT 分析
# - Leak Suspects: 自动分析泄漏嫌疑
# - Dominator Tree: 按占用排序
# - Path to GC Roots: 找到谁引用了它

# 4. 常见泄漏原因
# - 集合只加不删(静态 Map/List)
# - 连接未关闭(DB/HTTP/IO)
# - ThreadLocal 未 remove
# - 监听器未注销
```

### 2.3 接口响应慢排查

```
排查流程:
1. 确认范围: 所有接口慢 or 某个接口慢?
2. 看监控: Grafana/Prometheus 找拐点
3. 链路追踪: SkyWalking/Zipkin 定位慢在哪一层
4. 分层排查:
   - 网络层: ping/traceroute
   - 应用层: Arthas trace 方法耗时
   - DB层: 慢查询日志 + EXPLAIN
   - 缓存层: Redis SLOWLOG
   - 中间件: MQ 积压? RPC 超时?
5. 解决: 加缓存/优化SQL/异步化/扩容
```

### 2.4 死锁排查

```bash
# 1. jstack 找死锁
jstack <pid> | grep -i "deadlock"

# 2. Arthas
arthas> thread -b    # 直接打印阻塞线程

# 3. MySQL 死锁
SHOW ENGINE INNODB STATUS\G
# 看 LATEST DETECTED DEADLOCK 部分

# 解决:
# - 固定加锁顺序
# - 减小事务粒度
# - 加超时时间
```

## 三、高频场景题

### 3.1 如何保证数据一致性（缓存与DB）

```
方案: Cache Aside Pattern (旁路缓存)

写: 先更新 DB → 再删除缓存
读: 先查缓存 → 未命中查 DB → 写回缓存

问题: 删缓存失败怎么办?
方案:
1. 延迟双删: 更新DB → 删缓存 → sleep 500ms → 再删一次
2. 订阅Binlog: Canal 监听 → 异步删缓存(可靠)
3. 消息队列重试: 删失败 → 发MQ → 消费者重试删除
```

### 3.2 如何处理大数据量导出

```
问题: 导出 100 万条数据，内存溢出/超时

方案:
1. 流式查询: MyBatis 游标(Cursor) / fetchSize
2. 分页查询: WHERE id > lastId LIMIT 1000
3. 异步导出: 提交任务 → MQ → 后台生成文件 → 通知下载
4. 文件格式: 用 EasyExcel(SAX模式) 代替 POI
5. 分片: 大文件拆多个sheet或多个文件打包

关键:
- 不要一次性加载到内存
- 用流式写入(flush per batch)
- 设置超时兜底
```

### 3.3 如何处理消息积压

```
1. 紧急扩容: 增加 Consumer 实例(不超过 Partition 数)
2. 临时方案: 新建 Topic + 临时 Consumer 做消息搬运
3. 跳过非核心: 业务降级,只处理重要消息
4. 根治: 
   - 排查消费慢的原因(DB慢? 外部调用慢?)
   - 批量消费
   - 增加 Partition 数
```

### 3.4 如何设计一个对账系统

```
场景: 支付系统与第三方(支付宝/微信)对账

流程:
1. T+1 下载对账文件(SFTP/API)
2. 解析文件，存入对账临时表
3. 逐笔比对: 本地订单 vs 三方记录
4. 差异标记: 长款(我多他少) / 短款(我少他多)
5. 人工/自动处理差异
6. 生成对账报告

技术:
- 大数据量: 分批加载 + 多线程比对
- 幂等: 对账任务有唯一标识,防止重复执行
- 容错: 文件下载失败重试 + 告警
```

### 3.5 如何保证接口幂等

```
场景: 用户重复点击 / 网络重试 / MQ 重复投递

方案矩阵:
┌──────────┬─────────────────────────────────────┐
│ 场景     │ 方案                                │
├──────────┼─────────────────────────────────────┤
│ 创建订单 │ 唯一索引(订单号) / Token机制         │
│ 支付     │ 状态机(已支付则忽略)                 │
│ MQ消费   │ 消息ID去重(Redis SETNX)             │
│ 更新     │ 乐观锁(version字段)                 │
│ 通用     │ 请求ID + Redis 去重表(TTL 10min)    │
└──────────┴─────────────────────────────────────┘
```

### 3.6 如何做灰度发布

```
方案:
1. 按用户ID尾号: userId % 100 < 5 → 新版本
2. 按地域: 某城市先上
3. 按请求头: 特定 Header 走新版本
4. 按权重: Nginx upstream weight

技术实现:
- 网关层: Spring Cloud Gateway + 动态路由
- 注册中心: Nacos metadata 标签路由
- 全链路: 染色标记透传(TraceId 带灰度标)
```

## 四、项目亮点包装模板

### 4.1 性能优化类

```
【背景】XX 系统 XX 接口在高峰期 RT 达到 Xs，影响用户体验
【方案】
  1. 慢查询优化: 加联合索引，RT 从 500ms → 50ms
  2. 引入缓存: Redis + Caffeine 二级缓存
  3. 异步化: 非核心逻辑用 MQ 异步处理
  4. 批量合并: N+1 查询 → 批量查询
【结果】RT 从 Xs → Yms，QPS 提升 Z 倍
```

### 4.2 稳定性治理类

```
【背景】服务偶发超时导致上游级联故障
【方案】
  1. 熔断降级: Sentinel 配置熔断规则
  2. 超时治理: 统一 RPC 超时 + 重试策略
  3. 隔离: 线程池隔离核心/非核心接口
  4. 预案: 开关降级 + 限流兜底
【结果】故障恢复时间从 30min → 自动 30s，全年可用性 99.99%
```

### 4.3 架构升级类

```
【背景】单体应用代码量大、部署慢、互相影响
【方案】
  1. 领域划分: DDD 划分限界上下文
  2. 拆分策略: 按业务域垂直拆分
  3. 通信: Feign + MQ 解耦
  4. 数据: 分库分表(ShardingSphere)
  5. 灰度: 双写 → 校验 → 切流
【结果】部署频率从周级 → 日级，故障爆炸半径缩小 80%
```
